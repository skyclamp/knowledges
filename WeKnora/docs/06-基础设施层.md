# WeKnora 基础设施层设计分析

本文基于 WeKnora 当前代码实现，对基础设施层的文档解析、文本分块、检索工具、向量数据库集成、网络搜索、数据源连接器、配置、日志追踪、错误处理、文件存储、迁移与构建部署进行梳理。重点描述“代码已经实现了什么、如何接线、扩展点在哪里”，避免把 README 中的规划项当成既有能力。

## 1. 文档解析

### 1.1 支持的文档格式

WeKnora 的文档解析并不是单一实现，而是“Go 侧解析引擎注册 + Python DocReader 服务 + 可选 MinerU”的组合。

1. Go 侧 `docparser` 本地注册了四类引擎：
   - `builtin`：依赖 DocReader 服务，适合复杂格式。
   - `simple`：Go 原生处理简单文本、JSON、CSV、图片、音频。
   - `mineru`：Go 直接调用自建 MinerU 服务。
   - `mineru_cloud`：Go 直接调用 MinerU Cloud API。

2. Go 侧声明的格式覆盖：
   - `builtin`：`docx`、`doc`、`pdf`、`md`、`markdown`、`xlsx`、`xls`、常见图片格式。
   - `simple`：`md`、`markdown`、`txt`、`csv`、`json`、图片格式、音频格式 `mp3/wav/m4a/flac/ogg`。
   - `mineru/mineru_cloud`：`pdf`、`doc`、`docx`、`ppt`、`pptx`、图片格式。

3. Python DocReader 的内置注册表还提供：
   - `builtin`：`docx/doc/pdf/md/xlsx/xls/image`。
   - `markitdown`：`pdf/docx/doc/pptx/ppt/xlsx/xls/csv/md`。

因此从系统整体能力看，WeKnora 当前可处理 PDF、Word、Excel、Markdown、TXT、CSV、JSON、图片、音频，以及 URL/HTML 页面。PPT/PPTX 主要通过 MarkItDown 或 MinerU 路径支持，不走 Go 原生 `simple` 引擎。

### 1.2 解析器架构和接口设计

解析层的核心抽象是 `DocumentReader` 风格的远程读取接口与本地引擎注册表：

1. Go 侧通过 `GRPCDocumentReader` 和 `HTTPDocumentReader` 抽象 DocReader 通信协议。
2. 请求模型 `types.ReadRequest` 同时支持：
   - 直接上传文件字节。
   - 传入 URL 由远端抓取。
   - 指定 `parser_engine` 与 `parser_engine_overrides`。
3. 返回模型 `types.ReadResult` 统一输出：
   - `MarkdownContent`
   - `ImageRefs`
   - `Metadata`
   - `Error`

这一设计的关键价值在于：上游知识入库流程不需要关心底层是 Python 解析、Go 简单转换、还是 MinerU，只消费标准化 Markdown 与图片引用。

### 1.3 各格式的解析实现方式

#### 1.3.1 Simple 引擎

`simple` 引擎完全在 Go 内处理，适合轻量格式：

1. Markdown/TXT：直接透传为 Markdown 内容。
2. CSV：转换为 Markdown 表格。
3. JSON：递归切分为多个合法 JSON 代码块，尽量保持嵌套路径完整，避免生成碎片化 JSON。
4. 图片：包装成 Markdown 图片引用，并将原始字节装入 `ImageRefs`。
5. 音频：返回占位 Markdown，同时保留 `AudioData`，后续由 ASR 管道处理。

这条路径的目标是减轻 DocReader 负担，避免所有输入都经过 Python 服务。

#### 1.3.2 DocReader `builtin` / `markitdown`

Python DocReader 采用 `ParserEngineRegistry` 注册具体解析器类：

1. `DocxParser`/`Docx2Parser`：处理 Word 文档。
2. `PDFParser`：当前实现是责任链包装，优先尝试 MarkItDown 兼容路径。
3. `ExcelParser`：利用 `pandas` 遍历多 Sheet，将每行转为 `列名: 值` 的文本块。
4. `ImageParser`：返回图片 Markdown 与 base64 图片字节。
5. `MarkdownParser`：处理 Markdown。
6. `MarkitdownParser`：承接 Office、PDF、CSV、PPT 等更广泛格式。

DocReader 对外暴露 `Read` 和 `ListEngines` 两个 RPC。Go 侧会把本地引擎能力与远端引擎能力合并，远端如果声明了同名引擎，其文件类型和描述优先，以便后端新增引擎时无需同步改 Go。

#### 1.3.3 HTML / URL 解析

URL 文档走两条链路：

1. DocReader `parse_url` 使用 Python `WebParser` 获取网页内容并转换为 Markdown。
2. 应用侧还有 `internal/infrastructure/web_fetch`，用于 Agent 或聊天流水线安全抓取公开网页正文，强调 SSRF 防护与正文抽取。

前者偏“文档解析”，后者偏“工具抓取”。

### 1.4 与 docreader Python 服务的协作方式

Go 应用与 DocReader 的协作是标准的远程解析调用：

1. 启动时通过配置 `docreader.addr` 和 `docreader.transport` 选择 gRPC 或 HTTP。
2. gRPC 默认使用 `round_robin`，并根据 `MAX_FILE_SIZE_MB` 设置消息大小上限。
3. 读请求会把文件字节、文件名、URL、标题、解析引擎、引擎覆盖参数一并下发。
4. DocReader 把解析结果统一封装为 Markdown 和内联图片字节返回给 Go。
5. Go 再通过 `ImageResolver`、文件服务和知识入库流程完成图片落盘、Chunk 生成、向量索引。

这种分工让 Python 侧专注富文档格式兼容，Go 侧专注业务编排、权限、存储和索引。

### 1.5 OCR 处理

OCR 能力主要在 DocReader 和 MinerU 两侧：

1. DocReader 默认 OCR 后端是 PaddleOCR。
2. 支持 `OCR_BACKEND` 切换为：
   - `paddle`
   - `no_ocr`
   - `api`
3. PaddleOCR 强制走 CPU，且对 AVX 指令兼容性做了保护。
4. DocReader 还支持 VLM 配置，用于图像理解和多模态处理。
5. MinerU 自建与云版本都支持 OCR 开关、表格开关、公式开关、语言参数。

系统层面上，OCR 结果仍被统一回写为 Markdown 内容和图片引用，后续分块与索引流程不感知具体 OCR 来源。

## 2. 文本分块策略

### 2.1 支持的分块策略

当前代码里明确存在两种主策略：

1. 普通递归分块：`SplitText`。
2. 父子分块：`SplitTextParentChild`。

从实现看，没有单独的“语义模型分块器”服务；所谓语义保护主要通过“不可拆保护块”实现，而不是额外引入 embedding 语义切分算法。

### 2.2 分块参数

基础参数来自两处：

1. 全局配置 `knowledge_base`：
   - `chunk_size: 512`
   - `chunk_overlap: 50`
   - `split_markers: ["\n\n", "\n", "。"]`
2. 知识库级 `ChunkingConfig`：
   - `chunk_size`
   - `chunk_overlap`
   - `separators`
   - `enable_parent_child`
   - `parent_chunk_size`
   - `child_chunk_size`
   - `parser_engine_rules`

系统会校验 `chunk_overlap < chunk_size`，避免非法配置。

### 2.3 普通分块实现细节

`chunker.SplitText` 不是简单按字符截断，而是带规则保护的递归拆分器：

1. 优先按分隔符拆分，保留分隔符。
2. 保护不应被拆开的内容：
   - LaTeX 块公式
   - Markdown 图片
   - Markdown 链接
   - 表格头和表格行
   - 代码块
3. 对过大的保护块再做强制切分，避免下游 embedding 超限。
4. 合并阶段维护重叠窗口，并借助 `headerTracker` 让大表格跨块时继承表头。

这说明 WeKnora 的分块器是“文本结构感知”的，不是裸字符串切割。

### 2.4 父子分块机制

父子分块是当前代码里真实启用的高级策略：

1. 父块先按大窗口切分，默认 `parent_chunk_size = 4096`。
2. 再对每个父块切分子块，默认 `child_chunk_size = 384`。
3. 子块重叠固定为 `child_size / 5`，约 20%。
4. 子块记录 `ParentIndex`，入库后写入 `ParentChunkID`。
5. 父块 `ChunkType` 为 `parent_text`，进入数据库但不参与向量索引。
6. 子块 `ChunkType` 为普通 `text`，用于 embedding 和检索。

其设计目标很明确：

1. 检索时用较小子块提高召回精度。
2. 返回时可追溯父块，恢复更完整上下文。
3. 数据库迁移也为 `parent_chunk_id` 和 `chunk_type` 建了索引，说明这是持久化模型的一部分，不是临时内存策略。

## 3. 搜索与检索工具

### 3.1 全文搜索实现

WeKnora 的全文搜索不是单独一个服务，而是由各检索后端各自提供关键词检索能力：

1. PostgreSQL：通过 ParadeDB/全文检索函数做关键词匹配。
2. Elasticsearch：走 BM25 与查询 DSL。
3. SQLite：使用 FTS5 虚拟表 `lite_embeddings_fts`。
4. Qdrant：为 `content` 建 `text` 类型 payload index，使用多语言 tokenizer。
5. Milvus：为 `content` 建分析器与 BM25 稀疏向量字段 `content_sparse`。
6. Weaviate：类属性 `content` 使用 `text` 和 `gse` 分词，支持关键词检索。

因此“全文搜索”被抽象成统一的 `KeywordsRetrieverType`，但真正实现依赖后端引擎能力。

### 3.2 向量搜索实现

向量搜索通过统一接口 `RetrieveEngine` / `RetrieveEngineRepository` / `RetrieveEngineService` 组织：

1. `RetrieveParams` 包含查询文本、embedding、过滤条件、TopK、阈值、检索类型。
2. 向量索引创建由 `KeywordsVectorHybridRetrieveEngineService` 负责。
3. 批量索引时先批量生成 embedding，再分批并发写入各后端。

各后端实现方式：

1. PostgreSQL：`pgvector`，注释中使用半精度向量与 HNSW 索引思路。
2. Elasticsearch：`dense_vector` 与 script/query。
3. SQLite：`sqlite-vec` 的 `vec0` 虚拟表，按维度建表。
4. Milvus：按维度创建 collection，向量字段 + HNSW 索引。
5. Qdrant：按维度创建 collection，Cosine 距离。
6. Weaviate：按维度创建 class，vectorizer 为 `none`，由应用写入向量。

### 3.3 混合搜索策略

混合搜索入口在 `knowledgeBaseService.HybridSearch`：

1. 根据知识库配置和租户引擎配置，构造复合检索引擎。
2. 自动做 `5x` 过检索，保证后续融合与 rerank 有足够候选。
3. 同时构造：
   - `VectorRetrieverType`
   - `KeywordsRetrieverType`
4. FAQ 场景会有专门的后处理逻辑。

值得注意的是，混合检索是“应用服务层编排 + 后端引擎执行”的模式，而不是由某个数据库原生一把完成。

### 3.4 搜索结果融合算法

融合逻辑位于 `knowledgebase_search_fusion.go`，采用加权 RRF：

$$
score = \frac{0.7}{60 + rank_{vector}} + \frac{0.3}{60 + rank_{keyword}}
$$

设计要点：

1. 向量结果权重 0.7，关键词结果权重 0.3。
2. 若只有向量结果，则只做去重保留最高分。
3. 融合后按 RRF 分数降序排序。
4. 日志中会打印部分 top 结果用于调试。

这是一种稳定且工程化的融合方案，避免不同后端分数尺度不一致的问题。

### 3.5 Rerank 与检索增强

虽然本章重点是搜索工具，但当前检索链路实际把 rerank 作为检索增强的一部分：

1. Conversation 配置里提供 `enable_rerank`、`rerank_top_k`、`rerank_threshold`。
2. chat pipeline 中有独立 rerank 插件。
3. rerank 前会做段落清洗，过滤空段落、直接加载结果等。

这意味着 WeKnora 的检索并不是“向量 + 关键词”结束，而是可继续进入重排阶段。

### 3.6 繁简转换（OpenCC）支持

繁简转换的代码落点比较克制，主要用于标准化而不是全文预处理：

1. FAQ 相关逻辑使用 `github.com/longbridgeapp/opencc`。
2. 当前主要用于传统中文转简体，提高 FAQ 问题去重和哈希一致性。

因此它是“检索辅助标准化能力”，不是全局搜索前处理流水线。

## 4. 向量数据库集成

### 4.1 向量数据库抽象接口

向量库集成围绕三层接口展开：

1. `RetrieveEngine`：定义 `Retrieve()` 与 `Support()`。
2. `RetrieveEngineRepository`：扩展索引写入、删除、复制、批量状态更新等能力。
3. `RetrieveEngineService`：负责 embedding 生成、批量索引、删除与复制的业务编排。
4. `RetrieveEngineRegistry`：统一注册和获取具体引擎。

这套抽象的核心思路是：检索后端既承担查询，也承担索引存储生命周期管理。

### 4.2 已集成的向量数据库适配器

当前代码实际接入的引擎类型包括：

1. PostgreSQL / pgvector
2. Elasticsearch v7
3. Elasticsearch v8
4. Qdrant
5. Milvus
6. Weaviate
7. SQLite-Vec

类型常量定义在 `internal/types/retriever.go`，注册由容器初始化 `initRetrieveEngineRegistry` 完成。

#### 4.2.1 SQLite-Vec

1. 在 SQLite 主库内创建 `lite_embeddings` 元数据表。
2. 同时建立 FTS5 全文表和按维度拆分的 `vec0` 虚拟表。
3. 适合单机和轻量部署，是“数据库 + 向量检索”合一的 Lite 方案。

#### 4.2.2 Milvus

1. 按 embedding 维度创建不同 collection。
2. 向量字段建 HNSW。
3. 额外创建 `content_sparse` 和 BM25 function，兼顾全文与向量能力。
4. 度量类型由 `MILVUS_METRIC_TYPE` 控制，支持 `IP/COSINE/L2`。

#### 4.2.3 Weaviate

1. 按维度建 class。
2. 向量配置 `vectorizer=none`，由应用侧直接写入 embedding。
3. 内容字段启用 `gse` 分词，方便关键词检索。

#### 4.2.4 Elasticsearch

1. 统一索引名可由 `ELASTICSEARCH_INDEX` 指定。
2. v8 适配器会自动检测字段映射，决定是否给 ID 字段追加 `.keyword`。
3. 同一后端兼顾全文搜索和向量搜索。

#### 4.2.5 PostgreSQL / pgvector

1. 向量与元数据放在关系库中。
2. 关键词检索走 PostgreSQL/ParadeDB 全文能力。
3. 适合希望把事务性数据和检索数据放在同一数据库中的场景。

#### 4.2.6 Qdrant

1. 按维度拆分 collection。
2. 为常用过滤字段建立 payload index。
3. 为 `content` 建立多语言文本索引。

### 4.3 向量索引管理

索引生命周期由服务层统一编排：

1. 单条索引：`Index()`。
2. 批量索引：`BatchIndex()`，默认 40 条一批，最多 5 并发保存。
3. 删除索引：按 `chunk_id/source_id/knowledge_id` 删除。
4. 复制索引：支持知识库复制时免重新计算 embedding。
5. 批量更新启用状态与标签。

从工程角度看，WeKnora 不是把向量库当“只写一次”的黑盒，而是把它作为可维护索引层来管理。

### 4.4 `docs/使用其他向量数据库.md` 中的使用指南

仓库给出了清晰的扩展流程：

1. 实现 `RetrieveEngine`。
2. 实现 `RetrieveEngineRepository`。
3. 实现 `RetrieveEngineService`。
4. 增加环境变量配置。
5. 在 `initRetrieveEngineRegistry()` 注册。
6. 在 `internal/types/retriever.go` 增加类型常量。

这说明向量数据库扩展是“一等扩展点”，不是只能修改单点代码的硬编码实现。

## 5. 网络搜索引擎集成

### 5.1 搜索引擎提供商接口

网络搜索引擎使用 Registry + Factory 模式：

1. `web_search.Registry` 维护 `providerType -> factory` 映射。
2. 工厂函数签名统一为：`func(params types.WebSearchProviderParameters) (interfaces.WebSearchProvider, error)`。
3. Provider 运行时由租户自己的参数实例化，而不是用全局环境变量拼装。

这让多租户可以各自配置不同的搜索引擎凭证。

### 5.2 已集成的搜索引擎

当前已实现的 Provider 包括：

1. Bing
2. Google Custom Search
3. DuckDuckGo
4. Tavily

实现特征：

1. Bing/Tavily/Google 主要走官方 API。
2. DuckDuckGo 先走 HTML 抓取，再回退到 Instant Answer API。
3. Google 需要 `APIKey + EngineID`。

### 5.3 搜索结果处理

网络搜索结果并不会停留在外部结果结构体，而是可继续并入知识检索链路：

1. `types.WebSearchResult` 保存标题、URL、摘要、来源、发布时间。
2. `searchutil.ConvertWebSearchResults` 把它们转换成统一的 `SearchResult`。
3. 结果会填充 `KnowledgeSource=web_search`、`ChunkType=web_search` 等元数据。
4. 这样聊天流水线、Agent 工具和结果合并逻辑都可以复用已有的搜索结果结构。

### 5.4 扩展指南中的安全约束

`docs/添加新的网络搜索引擎.md` 明确强调了一个关键安全原则：

1. 搜索 API 的 BaseURL 必须硬编码。
2. 不向租户暴露可自定义的搜索目标域名。
3. 目的是从源头避免 SSRF。

这是当前网络搜索集成最值得关注的基础设施设计点之一。

## 6. 数据源连接器

### 6.1 连接器架构和接口

数据源同步框架采用标准 Adapter 模式：

1. `Connector` 接口统一定义：
   - `Type()`
   - `Validate()`
   - `ListResources()`
   - `FetchAll()`
   - `FetchIncremental()`
2. `ConnectorRegistry` 按类型注册与查找实现。
3. `DataSourceService` 负责 CRUD、凭证校验、资源发现、同步触发、内容入库。
4. `Scheduler` 负责定时调度。
5. 真正同步执行通过 asynq 队列或 Lite 模式同步执行器完成。

### 6.2 已实现的连接器

当前代码中真正完成实现的是飞书连接器：

1. 支持校验 App ID / App Secret。
2. 可列出 Wiki Space 作为可选资源。
3. 支持全量递归抓取空间节点。
4. 支持增量同步，依据 `obj_edit_time/node_edit_time` 检测变更。
5. 支持删除检测。

虽然元数据注册表列出了 Notion、Confluence、GitHub、Google Drive 等类型，但当前工作区里只有 Feishu 有完整 connector 实现，应在文档中明确区分“元数据预留”和“代码已实现”。

### 6.3 调度器设计

调度器使用 `robfig/cron` + `asynq` 的两层去重：

1. 启动时从数据库加载所有 active 数据源并注册 cron。
2. 触发时先查 `HasRunningSync`，防止上次任务未结束又重入。
3. 任务投递使用确定性的 `TaskID=dssync:<dataSourceID>:<minute>`。
4. 多实例下由 Redis 保证同一分钟只有一个实例成功入队。

因此它既能支持单机，也考虑了多实例下的分布式去重。

### 6.4 扩展新连接器的方式

`CONNECTOR_IMPLEMENTATION_GUIDE.md` 和 `docs/数据源导入开发文档.md` 给出的步骤很完整：

1. 新建 `client.go / connector.go / types.go`。
2. 实现配置解析与连接测试。
3. 实现资源发现。
4. 实现全量同步与增量同步。
5. 增量同步通过 `SyncCursor` 记录上次状态。
6. 在容器或注册表中完成注册。

凭证落库时采用 AES-256-GCM 加密，说明连接器框架不仅关注抓取，也关注多租户凭证安全。

## 8. 日志与可观测性

### 8.1 日志框架和级别

日志系统基于 Logrus 自定义封装：

1. 统一使用私有 `appLogger`，避免外部修改全局 logger。
2. 支持级别：`debug/info/warn/error/fatal`。
3. 日志输出统一到 stdout，适配 Docker 日志采集。
4. 支持彩色格式化，但仅在终端环境下启用。
5. 自动补充 `caller`、`request_id` 等字段。
6. `CloneContext()` 会复制关键上下文字段，方便异步链路延续日志上下文。

整体上，这是一套偏工程实用主义的结构化日志设计。

### 8.2 链路追踪实现

追踪系统基于 OpenTelemetry：

1. `InitTracer()` 初始化全局 TracerProvider。
2. 若设置 `OTEL_EXPORTER_OTLP_ENDPOINT`，则使用 OTLP gRPC exporter。
3. 未设置时回退到 stdout exporter。
4. 使用 `AlwaysSample()`，说明默认全采样，更适合开发或中低流量场景。
5. `ContextWithSpan()` 提供统一创建 span 的助手函数。

知识服务中的 `processChunks()` 等关键路径已经开始显式打 span，说明 tracing 不只是预埋，而是已进入部分核心流程。

### 8.3 指标收集

从当前代码状态看，指标收集是“部分具备基础、未形成统一观测面”的状态：

1. 运行时代码中没有看到集中式 Prometheus exporter 注册。
2. `internal/event/usage_example.md` 给出了 Prometheus 接入示例，但属于示例文档。
3. 代码中存在评估指标实现（Precision、Recall、MAP、MRR、NDCG），但这是离线/业务指标，不是服务监控指标。
4. IM 队列等少量模块暴露了内部 metrics 信息用于日志。

因此更准确的描述是：

1. 日志与 tracing 已较完整。
2. 监控指标仍以扩展点和示例为主，未形成统一内建指标平台。

## 9. 错误处理

### 9.1 错误类型定义

当前错误处理分为两层：

1. 通用业务错误：`AppError`
2. 局部哨兵错误：如 session 相关错误

`AppError` 包含：

1. `Code`
2. `Message`
3. `Details`
4. `HTTPCode`

这让 HTTP 层可以直接将业务错误映射为统一响应。

### 9.2 错误码规范

错误码采用分段管理：

1. `1000-1999`：通用错误
2. `2000-2099`：租户错误
3. `2100-2199`：Agent 错误

当前已定义的包括：

1. `ErrBadRequest`
2. `ErrUnauthorized`
3. `ErrForbidden`
4. `ErrNotFound`
5. `ErrConflict`
6. `ErrInternalServer`
7. `ErrValidation`
8. 若干租户和 Agent 配置错误

这种分段式约定利于后续扩展更多业务域错误码。

### 9.3 错误传播策略

从代码看，错误传播采用“返回 error + 边界统一包装”的策略：

1. 基础设施层尽量返回原始错误或带上下文的 `fmt.Errorf("...: %w", err)`。
2. 业务边界处将错误记录到日志。
3. HTTP 层通过 `AppError` 映射状态码与响应体。
4. 调度器等后台任务会把错误回写到 `sync_logs.error_message`。
5. 数据库迁移错误会给出修复建议，尤其是 dirty state 的 force/goto 提示。

这不是强侵入式异常体系，而是 Go 风格显式返回 + 层级包装。

## 总结

WeKnora 的基础设施层整体呈现出明显的“分层清晰、能力可替换、以接口和注册表为中心”的工程风格：

1. 文档解析通过 Go 编排、Python DocReader 和 MinerU 组合实现，既保留复杂格式兼容性，也保留轻量本地路径。
2. 文本分块器对表格、代码块、公式、图片链接做了结构保护，并已实装父子分块。
3. 检索层以统一接口抽象多个向量数据库和全文搜索后端，混合搜索采用加权 RRF 融合。
4. 网络搜索与数据源连接器都采用注册表和工厂模式，扩展点清晰。
5. 配置、日志、追踪、迁移、文件存储等基础设施能力已具备生产化特征。
6. 相比日志与 tracing，统一指标平台仍偏弱，更像预留扩展点而非已完成能力。

从当前实现成熟度看，WeKnora 的基础设施层已经不只是“工具集合”，而是一个可扩展的 RAG 平台底座。
